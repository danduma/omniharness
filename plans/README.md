## Findings deferred (audited, not yet planned)

These came out of the audit as real or plausible but were **not** written as
plans — each needs deeper specification or carries refactor risk that a
zero-context executor shouldn't take on blind. Listed so the next session can
pick them up:

- **Characterization tests for supervisor core** (`src/server/supervisor/observer.ts`
  ~1336 LOC, `worker-availability.ts`, queued-FIFO, credential bridging). High
  value as a safety net, but specifying meaningful assertions for a 1300-line
  state machine requires deep module understanding first. Should land BEFORE any
  mode-routing refactor (below). Effort M.
- **`error.surfaced` gap on generic queued-steering failures**
  (`src/server/conversations/queued-messages.ts:542-589`). MED confidence: the
  path emits execution events but only surfaces a typed user-facing error for
  empty-output in direct/commit mode. Needs an investigate-then-harden plan that
  confirms the exact uncovered branches against the spec in
  `docs/architecture/lifecycle-observability-and-testing.md`. Effort M.
- **Fire-and-forget `cancelAgent` can orphan bridge agents**
  (`src/server/conversations/send-message.ts:788,850`; also `runs/recovery.ts`,
  `supervisor/index.ts`). MED confidence; fix interacts with worker recovery and
  should be designed alongside the recovery path. Effort M.
- **Unbounded per-run/per-worker module caches** (`queued-messages.ts:51`,
  `output-store.ts:82`, `plans/readiness-pipeline.ts:49`,
  `artifacts/stream-metadata.ts:30`). Real slow RSS growth, but LOW severity for
  a local single-user tool. NOTE: `lastQueuedMessageCreatedAtByRun` participates
  in FIFO logical-clock ordering (see
  `docs/superpowers/learnings/2026-05-20-queued-fifo-needs-logical-clock.md`),
  so naive size-based eviction is unsafe — the right fix is cleanup-on-delete,
  which needs the conversation-delete path mapped first. Effort S–M.
- **Scattered mode-routing checks** (`mode === "direct"/"planning"/…` across 32
  files vs the `modes.ts` helpers). Real duplication, but consolidating routing
  on critical paths is HIGH-risk and should follow the supervisor
  characterization tests. Effort M.

## Direction (options for the maintainer, not ranked against bugs)

- **In-app "Improve mode"** is fully spec'd in
  `docs/superpowers/plans/2026-06-14-visual-improve-mode.md` but unbuilt (no
  `src/server/improve/`). Highest-grounded next feature; L effort, HIGH
  integration risk. Best entered as a spike plan (schema + parser + failing
  tests first).
- **Harden the remote-access surface.** README ships cloudflared remote access;
  there is no `middleware.ts` and no CSP/security headers. If remote access is a
  real use case, a bounded hardening pass (headers, optional always-`Secure`
  cookies) is grounded. Plan 002 reduces the same surface's known CVE exposure.
- **Pluggable conversation modes.** `modes.ts` is a hardcoded union; the
  mode-routing consolidation (deferred above) is the natural precursor to a mode
  registry. No evidence of user demand yet — spike only.

## Findings considered and rejected (so they aren't re-audited)

- **events.ts ships all workers regardless of selection** — false. `events.ts:115`
  already filters `inArray(workers.runId, visibleRunIds)`.
- **Duplicate queued-message delivery via drain race** — already mitigated; the
  drain claim is atomic (`update…where(and(eq(status,"pending")))`) and the
  learning `docs/superpowers/learnings/2026-06-16-queued-drain-claims-must-be-atomic.md`
  settled it.
- **events.ts linear `allRuns.find`/`.some`** — a handful of calls per snapshot
  over non-archived runs only; not worth a Map.
- **Cookie `Secure` only in production / `SameSite=Lax`** — by-design for a
  127.0.0.1 HTTP local tool; state-changing routes already enforce same-origin
  (`enforceSameOrigin: true`). Not a finding.
- **Account API DTO redaction** — already correct (`src/server/accounts/dto.ts`);
  matches `docs/superpowers/learnings/2026-06-13-account-api-dto-redaction.md`.
- **`vitest` UI RCE as a standalone item** — folded into plan 002's version bump;
  reachability is low because the repo runs `vitest run`, never the UI server.
- **Stale `OMNI_UNIFIED_WORKER_STREAM` flag doc** (`.env.example:44`,
  `docs/architecture/worker-conversation-stream.md:196`) — the flag is read
  nowhere in `src/`. Real doc-accuracy nit, but "remove the doc" vs "the flag may
  return" is a maintainer judgment, not an executor task. Left as a note here
  rather than a plan.

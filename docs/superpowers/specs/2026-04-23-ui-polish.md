# AI UI Elements Polish Design

## Goal
Improve the visual integration and readability of AI-built CLI Agent elements within the OmniHarness web UI. Specifically, shorten worker IDs to a more human-readable format, redesign the system message output (like "Spawned worker"), and refine the "Ready to retry" failure notice.

## Core Job
To provide users with clean, professional, and readable UI for managing and observing AI workers, eliminating raw logging clutter from the frontend experience.

## Stories
- As a user, I want to easily distinguish workers using sequential numbering per chat (e.g., Worker 1, Worker 2) instead of cryptic UUIDs or timestamps.
- As a user, I want system events like "Spawned worker" to look like intentional UI timeline events rather than raw log strings.
- As a user, I want the agent cards and recovery actions ("Ready to retry") to fit cohesively into the ShadCN-based design language of the application.

## Design

### 1. Worker ID Generation
- **Backend**: Update `src/server/supervisor/index.ts` to assign worker IDs sequentially scoped to the `runId`. Format: `<runId>-<increment>` (e.g., `123e4567-e89b-12d3-a456-426614174000-1`).
- **Frontend**: Update `WorkerCard` and other relevant UI elements in `src/app/page.tsx` to display the shortened name: `Worker <increment>` (e.g., `Worker 1`, `Worker 2`). The full ID can remain available in a tooltip (`title` attribute) for debugging.

### 2. System Messages Rendering
- **Parsing**: In `src/app/page.tsx`, intercept `msg.role === "system"`. If the `content` starts with `"Spawned worker."`, parse the `|` delimited string.
- **UI**: Render parsed spawn messages as structured timeline cards with an icon (e.g., `<Cpu className="text-primary h-4 w-4" />`), a bold title ("Worker 1 Spawned"), and metadata badges (CLI, Mode, Purpose) instead of raw mono-spaced text.
- General system messages should remain subtle but receive slightly more padded, rounded styling to feel like application events rather than console dumps.

### 3. Recovery Notice ("Ready to retry")
- The current implementation reuses an error-notice style for the "Ready to retry" state, resulting in a green border but potentially confusing layout.
- We will update the `conversationFailure` styling in `src/app/page.tsx` or the component rendering it to use a softer, more affirmative "Next Steps" / Actionable card style (e.g., using `bg-emerald-500/10 border-emerald-500/20 text-emerald-200` with an explicit action button or clearer layout).

## Architecture & Boundaries
- All changes are localized to `src/server/supervisor/index.ts` (ID generation) and `src/app/page.tsx` (presentation layer).
- Backend database schema (`workers.id` as `TEXT`) remains unchanged and compatible with the new `<runId>-<increment>` format.

## Acceptance Criteria
- Workers spawned during a run are identified sequentially (1, 2, 3...) on the frontend.
- System messages for "Spawned worker" parse the string and render a polished UI component.
- The "Ready to retry" notice does not look like an error state, but rather a clear, actionable path forward.

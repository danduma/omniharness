# Direct Active Streams Need Stop Targets And Revalidation

**Date:** 2026-05-20
**Context:** OmniHarness direct-control conversations, composer controls, unified worker stream
**Symptom:** A direct session could show "Thinking..." with no stop button. Another direct follow-up had worker output persisted on disk, but the selected frontend terminal stayed empty until a force reload or session switch.
**Root Cause:** The UI used different evidence for "active" and "stoppable": direct `running` state could render the pending assistant indicator without producing a worker id for `stop_worker`. Separately, `WorkerEntriesManager.ensureLoaded()` treated a stream that was caught up to the last known seq as complete until an SSE wake-up or snapshot cursor arrived, so a missed wake-up left the visible stream stale.
**Fix:** Direct running state now falls back to the selected direct worker as the stop target. The composer shows a separate stop button whenever the conversation is stoppable but the submit button is being used for send/queue/steer. The selected direct worker stream can explicitly refresh from its current contiguous cursor; `ConversationMain` now uses the faster refresh while the pending assistant indicator is visible and a slower validation refresh while the selected direct stream appears idle.
**Verification:** Added failing regression coverage first, then passed `pnpm vitest run tests/app/worker-entries-manager.test.ts tests/ui/composer-shell.test.ts` with 27 tests passing. Also passed `pnpm vitest run tests/app/worker-entries-manager.test.ts tests/ui/composer-shell.test.ts tests/app/direct-worker-stream-loading.test.ts tests/app/busy-message-behavior.test.ts tests/api/conversation-messages-route.test.ts` with 60 tests passing, `pnpm vitest run tests/server/git/auto-commit.test.ts` with 5 tests passing, `pnpm lint` with 0 errors and existing warnings, and `git diff --check`.
**Prevention:** Any UI that says a direct conversation is working must also prove a stoppable worker id or explicitly explain why stopping is impossible. Any selected direct stream must have a deterministic revalidation path that does not depend on session switching, reloads, or perfectly timed SSE.
**Skill/Doc Updates:** Updated `docs/architecture/timing-determinism-audit.md` and `docs/architecture/direct-control-session-regressions.md` because this is a project-specific recurrence of the direct-control race class; no global skill update was needed beyond applying the existing client/server state invariant rules.

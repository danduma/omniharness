# Stale Worker Session Recovery

**Date:** 2026-05-30
**Context:** OmniHarness direct/planning worker lifecycle recovery and resource admission.
**Symptom:** A worker spawn refused for low disk space, then a later retry surfaced `Direct retry could not locate persisted ACP session metadata for this conversation`, even though the user action was simply to resume after freeing disk.
**Root Cause:** The recovery path treated missing ACP session metadata as a fatal user-visible resume error instead of falling back to a fresh direct rerun. Separately, stale `starting`/`working` DB rows could remain active after the bridge no longer had a worker, so the UI could keep offering recovery paths backed by dead runtime state. The resource refusal copy also always suggested waiting for other workers, even for disk-only failures where no workers might be running.
**Fix:** Direct retry now records a diagnostic event and falls through to fresh-worker retry when saved session metadata is absent. Reload reconciliation now marks old active workers missing from the bridge as `lost`/`needs_recovery`, preflights saved session metadata before attempting auto-resume, and avoids surfacing a resume banner when no session exists. Resource admission copy now names disk or memory pressure directly and only mentions other workers conditionally.
**Verification:** `pnpm vitest run tests/server/runs/persisted-zombie-reconciler.test.ts`; `pnpm vitest run tests/server/conversations-sync.test.ts tests/server/agent-runtime/manager-reaper.test.ts`; `pnpm vitest run tests/api/run-route.test.ts tests/api/conversations-route.test.ts --pool forks --poolOptions.forks.singleFork`; `pnpm exec tsc --noEmit`.
**Prevention:** Treat persisted worker rows as stale unless the bridge or durable worker stream can prove recoverability. Never surface missing ACP session metadata as the first-class user error when a fresh direct retry can proceed. For lifecycle tests with background turns, wait for tracked background tasks and clear artifact stream metadata/caches before DB cleanup.
**Skill/Doc Updates:** No general skill update needed; the reusable project-specific rule is captured here and the lifecycle spec already requires explicit recovery transitions and user-visible failure codes.
